## What this repository is

This repository holds both the Portuguese-language PRD (Product Requirements Document, chapters 1–28, complete) for the "Nexus Platform" and, as of Fase 1 (Cap. 28), the actual Next.js source code implementing it. The `.md` PRD files live alongside the app at the repo root — this is intentional (see "Resolved decisions" below: user chose to scaffold in-place rather than in a subdirectory). There is still **no git repository initialized** (also an explicit choice — init it when the user asks).

Fase 1 ("Estrutura Base") is done: Next.js (App Router, TS, Tailwind v4, Turbopack) + shadcn/ui + Motion + Lucide + ESLint + Prettier + Vitest are installed and configured, the `src/` folder structure matches Capítulo 18 (with the resolved decisions applied), and `next build` / `npm run lint` / `npm run test` all pass.

Fase 2 ("Design System") is done for the base component set: Button, Input, Card, Badge, Avatar, Dialog (= PRD's "Modais"), Tooltip, DropdownMenu, Tabs, Accordion, Skeleton, Toast (`sonner`), Sidebar, NavigationMenu are installed via shadcn in `src/components/ui/`. Button is restyled to the Capítulo 17 spec (see below).

Fase 3 ("Layout Global") is done: `Navbar` (`src/components/navigation/Navbar.tsx`) with the Capítulo 19 mega menu ("Soluções"), dropdown ("Empresa"), transparent→scrolled state via `useScrolled`, and a mobile fullscreen `Sheet` menu; `Footer` (`src/components/layout/Footer.tsx`) with the Capítulo 20 six-column layout, socials, and bottom bar; `Container` (`src/components/layout/Container.tsx`) implementing the Capítulo 17 grid (1280px max-width, 24/64/120px responsive padding). Both are wired into `src/app/(marketing)/layout.tsx`, applied only to the `(marketing)` route group — `(auth)` and `(portal)` will get their own layouts (split-screen / sidebar, Cap. 21) in later phases, not this Navbar/Footer chrome.

Fase 4 ("Home") is done: all 7 sections are built as features under `src/features/` and composed in `src/app/(marketing)/page.tsx`, in the official order — `Hero` → `Ecosystem` → `Capabilities` → `PortfolioSection` → `Process` → `InteractiveDemo` → `Cta`. Each feature folder holds `constants.ts` (copy/data) + the component(s) + `index.ts` barrel; sub-components (e.g. `EcosystemNode`, `DemoNode`, `HeroMockup`) stay uncombined with the 6-file Cap. 28 mold per the resolved decision (they have real state, but splitting a 40-line component into 6 files would be pure ceremony). Notable implementation choices:

- **Hero** (`features/hero/`) uses the newly-installed React Bits `DotGrid` (`components/backgrounds/`) + `SplitText` (`components/animations/`) for the background and headline animation, plus a hand-built floating/rotating dashboard mockup (`HeroMockup.tsx` — no real product screenshot exists, so this is an illustrative CSS/Motion mockup, not a placeholder image). `Hero` uses `-mt-20` to counteract the `(marketing)/layout.tsx` `pt-20` and sit behind the transparent Navbar as Capítulo 11 specifies.
- **Ecosystem** and **InteractiveDemo** (`features/ecosystem/`, `features/interactive-demo/`) both use Magic UI's `AnimatedBeam` (`components/ui/animated-beam.tsx`, Motion-based, no GSAP) to draw connections between DOM-ref'd node cards. `Ecosystem` highlights beams touching the hovered node; `InteractiveDemo` runs a scripted idle→processing→success sequence via `setTimeout` chains (cleared on unmount/restart) triggered by "Executar demonstração."
- **Portfolio** teaser (`features/portfolio/`) uses **placeholder project data** (`constants.ts`) matching Capítulo 13's "Projetos Iniciais" list (CRM Nexus, Barberflix, etc.) with a gradient-block cover in place of real screenshots — this gets replaced by the CRM-driven API from Capítulo 25 in Fase 9, not before.
- **Capabilities** grid uses the resolved 3rd-row order (Integrações | Dashboards); the "Integrações" and "Dashboards" card copy was synthesized from capability fragments mentioned elsewhere in the PRD, since Capítulo 14's own "Cards previstos" list doesn't give "Integrações" its own dedicated sub-item set.
- **Cta** section has no dedicated PRD chapter (Cap. 1 only names it in the section list) — copy was written to match the established brand voice, not lifted from a chapter.

Fase 5 ("Páginas Institucionais") is done: `/solutions`, `/about`, `/contact`, `/portfolio` (full grid), `/portfolio/[slug]` (project detail, statically generated via `generateStaticParams`), `/privacy-policy`, `/terms-of-use`, and a 404 are all live. Notes:

- **Portfolio** data (`features/portfolio/constants.ts`) was extended from the Fase 4 teaser shape to the full Capítulo 18 contract (`slug, title, category, technologies, summary, challenge, solution, results`) and is now shared between the Home teaser, the full `/portfolio` grid, and the detail page via a common `ProjectCard` component — don't let these drift back into separate copies.
- **Solutions** page reuses `CAPABILITY_CARDS` (Fase 4) in a full-detail layout with `id` anchors (`#software`, `#ai`, etc.) matching the Navbar mega menu's hrefs — if you add/reorder capability cards, keep the `id`s in sync with `config/navigation.ts`'s `SOLUTIONS_MEGA_MENU`.
- The **"Processo" nav link** (Navbar "Empresa" dropdown + Footer) points to `/#process`, not `/about#process` — Process is a Home section (Fase 4), not embedded in the About page, despite Capítulo 19's link naming suggesting otherwise. `Process.tsx`'s `<section>` carries `id="process"` for this.
- **About** page content is Capítulo 1's Missão/Visão/Valores, verbatim — that chapter is the only PRD source for this page.
- **Contact form** (`features/contact/ContactForm.tsx`) has full client-side validation and all required states (default/hover/focus/loading/error/success/disabled per Cap. 24/27), but submission is simulated (`setTimeout` + success toast) — there's no `services/contact/` yet (Fase 6+), so don't assume this actually sends anything.
- **Privacy Policy / Terms of Use** are intentionally-labeled draft/template legal text (visible warning banner on each page) with `robots: { index: false, follow: false }` — don't remove that banner or flip them to indexable without an actual legal review; this was a deliberate choice, not an oversight.
- **404 needs two files, not one**: `(marketing)/not-found.tsx` only renders for unmatched paths _within_ an already-matched segment tree (e.g. `/portfolio/bogus-slug`, since `[slug]` is in `(marketing)`). A fully arbitrary top-level path (e.g. `/this-doesnt-exist`) never enters the `(marketing)` group at all and falls back to Next's generic default 404 unless a root `src/app/not-found.tsx` also exists — it does now, and manually renders `Navbar`/`Footer` itself (outside `(marketing)/layout.tsx`, so they aren't automatic there). Both delegate to the shared `components/layout/NotFoundContent.tsx` — edit that, not the two wrappers, to change 404 content/copy.

Fase 6 ("Autenticação") is done: Login, recuperação de senha, middleware, sessão, logout, and a basic Perfil page all work against a **real Supabase project** (credentials in `.env.local`, gitignored — see `.env.local.example` for the shape). Notes:

- **`services/supabase/{client,server,middleware}.ts`** wrap `@supabase/ssr`'s `createBrowserClient`/`createServerClient` per Supabase's official Next.js pattern. **Import each directly** (`@/services/supabase/client` or `@/services/supabase/server`), never through a shared barrel — an `index.ts` re-exporting both broke the build (`server.ts`'s `next/headers` import leaked into client bundles via the barrel graph even though only the browser export was used). If you add another Supabase helper, keep client-only and server-only code in separate files/imports, not a combined barrel.
- **`src/middleware.ts` doesn't exist — it's `src/proxy.ts`.** Next.js 16 renamed the convention (file _and_ the exported function, `middleware` → `proxy`); the old name still half-works but logs a deprecation warning at build time. `PRIVATE_ROUTES` from `config/navigation.ts` drives which paths it protects — keep using that single source of truth, don't hand-list routes here.
- **Root layout (`app/layout.tsx`) deliberately does not fetch the user server-side.** An earlier version called `cookies()`-dependent Supabase code there, which opted the _entire site_ out of static rendering (every public marketing page went from `○ Static` to `ƒ Dynamic` in the build output) — a real regression against Cap. 23. `SessionProvider` is now always seeded with `initialUser={null}` and resolves the real state client-side via `onAuthStateChange` (which supabase-js fires immediately on subscribe, then again on every login/logout). `(portal)/layout.tsx` still does its own server-side `getUser()` — that's fine, those routes are already inherently dynamic (per-request session check), unlike the public site.
- **Navbar shows a `Skeleton` avatar while `useSession().loading` is true**, never flashes "Entrar" first — matches Cap. 21's "nunca tela branca" loading requirement.
- **"Lembrar de mim"** is implemented by creating a one-off, non-singleton `createBrowserClient` (`createSignInClient(rememberMe)` in `services/supabase/client.ts`) with `cookieOptions.maxAge` set to 7 or 30 days _only for the sign-in call itself_ — every other client in the app just reads whatever cookie is already there. Don't switch `LoginForm` back to the shared singleton client or this breaks.
- **Password reset** (`(auth)/reset-password`) handles three possible arrival shapes defensively — PKCE `?code=`, `?token_hash=&type=recovery`, or implicit-flow hash auto-detected by the SDK — since which one Supabase actually sends depends on the project's Auth email-template/flow-type settings (dashboard config, not something this codebase controls).
- **`/dashboard` and `/profile` were minimal stubs at the time** — since superseded, see the "Portal retired, real SSO" note below; those routes don't exist in this codebase at all anymore.
- No sign-up page exists — Cap. 21 hedges account creation as "Criar Conta (caso liberado)," and access reads as admin-provisioned rather than public self-service, so this wasn't built by default.
- Verified against the real project directly: `curl` against `{SUPABASE_URL}/auth/v1/token?grant_type=password` with bad credentials returns `{"error_code":"invalid_credentials", ...}`, confirming both the credentials and `getAuthErrorMessage()`'s error-code mapping in `features/authentication/constants.ts` are correct.

Fase 7 ("Portal Nexus") originally built a `Sidebar`, a mock widget-rich `/dashboard`, `/profile`, and `/settings` — **all of this was later deleted**. See "Portal retired, real SSO with the CRM" below for the current state; don't go looking for `src/components/navigation/Sidebar.tsx` or `src/features/dashboard/`, they no longer exist.

**Fase 8 ("CRM") — history of three superseding decisions, ending in a full merge.** It was never built directly in this codebase; the user had a separate, fully-built CRM app (`nexus-prospect`), first linked out as an external, separately-authenticated destination (`target="_blank"`, no SSO), then (post-Fase 14, once a shared domain existed) upgraded to real cross-subdomain SSO via a shared Supabase session cookie — both of those are now purely historical, kept below for context. **The current, actual architecture**: the CRM's entire codebase was merged into this repository as a real path, `src/app/portal/` — single app, single Vercel deploy, single Supabase session, no cross-domain cookie needed at all. See "CRM merged into this app at /portal" further below (after the two historical sections) for what's actually true today.

### Portal retired, real SSO with the CRM (post-Fase 14) — historical, itself superseded by the full merge below

After Fase 14 (Deploy) shipped, the user clarified that the Fase 7 Portal (`/dashboard` with 100% mock data, `/profile`, `/settings`) no longer served any purpose — their external CRM already covers all of that for real. At the same time, they bought the domain **`teamnexus.com.br`** (this site at the root, the CRM at `crm.teamnexus.com.br`) and confirmed the CRM already runs on Supabase Auth too, just a different project. That combination made **real SSO** — not just an external link — both desirable and technically achievable, so the whole Portal was retired in favor of it:

- **Deleted entirely**: `src/app/(portal)/` (layout, loading, `dashboard/`, `profile/`, `settings/`, plus the already-empty `admin/`/`developer/` stub folders), `src/features/dashboard/` (all mock widgets/constants), `src/components/navigation/Sidebar.tsx`. None of these exist anymore — don't recreate them without the user asking again.
- **`PRIVATE_ROUTES` (`config/navigation.ts`) is now `[]`** — there's no local protected route left in this codebase for `proxy.ts`'s middleware to guard. The array is kept (not deleted) so the mechanism is ready the moment a real local private route exists again; `isPrivateRoute()` just always returns `false` in the meantime, which is safe (verified: `.some()` over `[]` doesn't throw, and `robots.ts`'s `disallow: [...PRIVATE_ROUTES]` becomes a valid empty disallow list).
- **SSO mechanism: shared, domain-scoped session cookie — not a token handoff.** Both this site and the CRM point at the **same Supabase project** (the user migrated this site to the CRM's project, not the other way around, since the CRM already had real users) and both set `cookieOptions: { domain: '.teamnexus.com.br' }` on every Supabase client (`services/supabase/client.ts`'s `createClient()`/`createSignInClient()`, `server.ts`, `middleware.ts` — all four call sites need this consistently, or the session cookie silently reverts to host-only on whichever one is missing it). Because `teamnexus.com.br` is the registrable domain (eTLD+1) for both `teamnexus.com.br` and `crm.teamnexus.com.br`, a plain navigation between them is same-site for SameSite cookie purposes — the browser sends the shared cookie automatically, no token in a URL, no receiver route needed on the CRM side.
- **The cookie domain is gated behind `NEXT_PUBLIC_COOKIE_DOMAIN`** (new env var, `services/supabase/env.ts`'s `getCookieDomain()`), unset by default. It must stay **empty on localhost and Vercel Preview deployments** — a page can't set a cookie for a domain that doesn't match its own host, so setting this anywhere but Production would just break login there. Only set it to `.teamnexus.com.br` in Vercel's Production environment, and only once both `teamnexus.com.br` and `crm.teamnexus.com.br` are actually attached as custom domains with valid certs — until then this whole mechanism is inert (the var is unset, so `cookieOptions.domain` resolves to `undefined`, which `@supabase/ssr`/the underlying `cookie` package treats as "omit the attribute," i.e. today's original host-only cookie behavior).
- **`LoginForm.tsx` no longer redirects to `/dashboard`.** On successful `signInWithPassword`, it does `window.location.href = CRM_URL` — a full page navigation (required, since `crm.teamnexus.com.br` is a different origin the Next router can't cross), relying entirely on the shared cookie to carry the session over, not on anything passed in the URL. The old `?next=` query-param redirect logic was removed along with it (dead code once `PRIVATE_ROUTES` is empty, and a redirect target read unvalidated from a query string was an open-redirect smell not worth keeping around for no remaining use).
- **`(auth)/login/page.tsx` is now a Server Component with an already-authenticated guard** — `if (user) redirect(CRM_URL)` — so a logged-in user hitting `/login` directly (e.g. a stale bookmark) gets bounced straight to the CRM instead of seeing the form again. Deliberately scoped to `/login` only, not the whole `(auth)/layout.tsx` — `/forgot-password`/`/reset-password` must stay reachable even mid-recovery for an authenticated user. Its copy also dropped the now-inaccurate "Acesse o Portal Nexus" wording in favor of "Acesse sua conta para continuar."
- **`ProfileDropdown.tsx` and `Navbar.tsx`'s mobile menu** both collapsed to just email + "Ir para o CRM" (`<a href={CRM_URL}>`, same-tab now — it's a first-class destination, not an external reference anymore) + "Sair." `CRM_URL` itself changed from `https://nexus-prospect-zeta.vercel.app/login` to `https://crm.teamnexus.com.br` (no `/login` suffix — points at the CRM's authenticated landing area, since a shared-session visitor shouldn't need its login page at all).
- **The CRM side needs matching changes in its own repo** (out of reach from this codebase): the same `cookieOptions: { domain: ... }` on every Supabase client it constructs, its Vercel project's custom domain set to `crm.teamnexus.com.br`, and confirmation it's pointed at the same Supabase project. No new route/endpoint is needed there — this is a much smaller ask than the token-handoff design that was considered and discarded during planning.
- **Supabase project swap is a deliberate, separate deploy step** — because `NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_SUPABASE_ANON_KEY` are inlined into the client bundle at build time, flipping them force-logs-out every session active at that moment. Do it in a low-traffic window, after confirming real accounts already exist in the CRM's project (there's no dual-project fallback once this ships).
- **Domain cutover (`NEXT_PUBLIC_SITE_URL=https://teamnexus.com.br`, `NEXT_PUBLIC_COOKIE_DOMAIN=.teamnexus.com.br` in Vercel Production) is a prerequisite outside of code** — DNS + Vercel custom domain for both projects. Until that's live, everything above still works except the actual cross-subdomain cookie sharing (login still works locally/on the current `*.vercel.app` URL, "Ir para o CRM" just won't carry the session yet).

### CRM merged into this app at /portal (supersedes both sections above — current state)

The cross-subdomain SSO approach above was itself superseded: the user decided to go further and bring the CRM's entire codebase (`nexus-prospect`, "Nexus Prospect System") into this repository as a real path prefix, `src/app/portal/` — one Next.js app, one Vercel deployment, one Supabase session, no cross-domain cookie needed at all. `crm.teamnexus.com.br` is no longer used by this app.

- **`src/app/portal/`** mirrors the CRM's original route tree: `dashboard`, `crm` (Kanban), `conversations` + `[leadId]`, `campaigns` + `[id]` + `new`, `financeiro`, `portfolio`, `prospecting`, `sdr`, `settings` (+ `openai`/`whatsapp`/`whatsapp-cloud`/`calendar`/`users` sub-pages, own `layout.tsx`), `calendar-test`. `src/app/portal/page.tsx` just `redirect("/portal/dashboard")`. `src/app/portal/layout.tsx` does the auth guard (server-side `getUser()`) and renders the Portal's own sidebar shell — same pattern the old Fase 7 `(portal)/layout.tsx` used, before it was deleted.
- **`PRIVATE_ROUTES` (`config/navigation.ts`) is `["/portal"]`**, not `[]` anymore — `isPrivateRoute()`'s existing prefix-match logic already protects everything under `/portal/*` from this one entry. `CRM_URL` no longer exists as a constant anywhere in the codebase.
- **`src/app/api/`** holds the CRM's ~44 API routes (webhooks for Evolution/WhatsApp Cloud, OpenAI audit/SDR, Google Calendar OAuth, campaigns, leads, financeiro, portfolio, prospecting, admin invite, etc.), copied with their exact original sub-paths preserved — several are webhook URLs already registered in Google Cloud Console / Meta App Dashboard / Evolution API instances, and must not move.
- **`src/services/portal/`** holds the CRM's business logic (`evolution/`, `google/`, `openai/`, `orchestration/`, `prospecting/`, `queries/`, `repositories/`, `sdr/`, `whatsapp-cloud/`, `settings.ts`) — this repo's `services/` = external API clients + data access convention, replacing the CRM's flatter `lib/`.
- **`src/features/portal-{crm,conversations,dashboard,campaigns,financeiro,portfolio,prospecting,sdr}/`** holds the CRM's domain components, one feature folder per module, matching this repo's `features/<name>/` convention instead of the CRM's `components/<domain>/` layout. File names were kept kebab-case (not renamed to this repo's usual PascalCase for components) — deferred as cosmetic polish, not required for correctness. `src/components/layout/portal-{sidebar,page-header}.tsx` hold the shell chrome (not a "feature" — no business logic of its own).
- **`services/supabase/server.ts` gained `createServiceClient()`** — same shape as the existing `createClient()`, but authenticated with `SUPABASE_SERVICE_ROLE_KEY` (new required env var) instead of the anon key, for the Portal's RLS-bypassing routes (webhooks writing on behalf of a user session that isn't present, `admin/invite`). Route Handlers only, never a Server/Client Component.
- **Auth is now fully unified**: `LoginForm.tsx` does `router.push("/portal/dashboard")` (same-origin client nav, not `window.location.href`) on success; `(auth)/login/page.tsx`'s already-authenticated guard redirects to `/portal/dashboard` too; `ProfileDropdown`/`Navbar`'s mobile menu link to `/portal/dashboard` via a normal `<Link>`. The CRM's invite flow (`admin.inviteUserByEmail`) and PKCE code-exchange both got ported into this app's `(auth)` route group as `/callback` (thin, redirects to `/set-password` for `type=invite`, else `/portal/dashboard`) and `/set-password` (reuses `ResetPasswordForm`'s logic via a new `SetPasswordForm.tsx`, rather than the CRM's differently-styled card, to stay visually consistent with the rest of `(auth)`). **A real bug in the CRM's original `proxy.ts` was fixed for free during this port**: it used to redirect any already-authenticated user away from `/set-password` straight to `/dashboard`, which would break the invite flow (since `/callback` authenticates *before* landing on `/set-password`) — this site's `proxy.ts` never had that pattern, so the merge just doesn't reproduce the bug.
- **Design tokens**: the site's existing hex-based Capítulo 17 tokens stayed the source of truth — the CRM's OKLCH chart palette was **not** imported; `--chart-1..5` were instead remapped onto the site's existing primary/success/warning/destructive/muted-foreground hex values. The `--sidebar-*` tokens the Portal shell needs already existed in `globals.css` as unused shadcn scaffold leftovers (Fase 11 dead-code note) — now actually used for the first time. Geist Mono (`next/font/google`) was added alongside the already-loaded Geist+Inter, wired to `--font-mono`, for the Financeiro/Dashboard tabular numbers.

**shadcn primitive library — read before building any interactive component**: this project's `shadcn init` picked the `base-nova` style, built on **Base UI** (`@base-ui/react`), not Radix. This changes real APIs you'll otherwise get wrong from Radix-based shadcn knowledge/muscle memory:

- No `asChild` prop. Use `render={<YourElement />}` (or a render-function) instead — e.g. `<DialogTrigger render={<Button>Abrir</Button>} />`, not `<DialogTrigger asChild><Button>...</Button></DialogTrigger>`. Passing `asChild` leaks it onto the DOM node as an invalid attribute (React will warn, not error — easy to miss). For internal Next.js routes, prefer `render={<Link href="..." />}` over a plain `href` prop so navigation stays client-side (`NavigationMenuLink`/`Button`/`SheetClose` all accept `href` directly too since they render `<a>`/native elements, but that forces a full page reload).
- Accordion has no `type="single"|"multiple"` + `collapsible` props. It takes `multiple?: boolean` (default `false`) and `value`/`defaultValue` as an **array**, e.g. `<Accordion defaultValue={["item-1"]}>`.
- `NavigationMenu` (used for the mega menu) needs no `viewport` prop — unlike Radix, the Base UI version always renders its shared positioner/viewport internally.
- When in doubt, check the type at `node_modules/@base-ui/react/<component>/**/*.d.ts` rather than assuming Radix conventions.

Some (not all) generic icons are also exported twice, once bare and once `*Icon`-suffixed (e.g. `Menu`/`MenuIcon`) — either works, shadcn's own generated files use the suffixed form.

**`react-hooks/refs` false positive**: passing `ref={someRefsObject.someKey}` (dot-notation property access on a plain object of `useRef()` values, built inline in the component and read directly in JSX) trips the `react-hooks/refs` ESLint rule ("Cannot access ref value during render"), even though this is just selecting which ref to forward, not reading `.current`. Bracket-notation access on the same kind of object _inside a `.map()`_ (e.g. `nodeRefs[node.id]`, as used in `Ecosystem.tsx`/`InteractiveDemo.tsx`'s beam connections) does not trigger it. When placing a handful of ref'd elements individually (not mapped), just use the individual `useRef()` variables directly instead of routing through a lookup object — see `InteractiveDemo.tsx`.

## Design system (Capítulo 17)

- Style: "Dark Premium" — high whitespace, minimal text, interface-forward, never decorative-for-its-own-sake.
- Grid: desktop reference width `1440px`, container `max-width: 1280px`. Padding: desktop `120px`, tablet `64px`, mobile `24px`.
- Spacing scale (8px-based, never arbitrary values): `4 8 16 24 32 40 48 64 80 96 120 160`.
- Radius scale: small `8px`, medium `16px`, large `24px`, hero `32px` — implemented as Tailwind utilities `rounded-sm/md/lg/xl` via `--radius-*` in `src/app/globals.css` (superseding Capítulo 1's single `18px` value, per resolved decisions above).
- Borders: always `1px solid rgba(255,255,255,.08)`. Shadows always very light — never heavy.
- Colors: background `#09090B`, surface `#111113`, surface hover `#17171A`, primary `#6C63FF`, primary hover `#7A73FF`, secondary `#8B5CF6`, success `#22C55E`, warning `#F59E0B`, danger `#EF4444`. Text: primary `#FFFFFF`, secondary `#A1A1AA`, muted `#71717A`.
- Typography: Geist primary / Inter fallback (`next/font/google`, wired in `src/app/layout.tsx`). Scale — hero `72px`, título `48px`, subtítulo `24px`, texto `18px`, descrição `16px`, legenda `14px` — implemented as Tailwind utilities `text-hero/title/subtitle/body/description/caption` via `--text-*` in `src/app/globals.css`.
- All colors above are implemented as CSS custom properties in `src/app/globals.css` (`:root`/`.dark`, kept identical since the PRD never specifies a light mode) and consumed through standard shadcn/Tailwind tokens (`bg-background`, `text-muted-foreground`, `bg-primary`, etc.) — never hardcode hex values in components.
- Buttons: height `48px`, radius `14px`, horizontal padding `24px`. Primary = filled with primary color + white text + subtle hover grow/shadow. Secondary = transparent + border, surface-hover on hover. Implemented in `src/components/ui/button.tsx` — `size="default"` carries the spec; `variant="default"` is primary, `variant="outline"` is the PRD's "Botão Secundário" (not shadcn's `variant="secondary"`, which is a separate filled-accent style using the `#8B5CF6` token, not one of the two PRD button types). Has an `isLoading` prop for the loading state.
- Cards: consistent structure (icon → title → description → action) — never a bespoke layout without justification.
- Icons: Lucide exclusively, sizes `20 / 24 / 32` — never mix icon libraries.
- Motion: entrance = fade + translateY 16px; hover = scale 1.02; click = scale 0.98; durations `200–400ms`, **never above 600ms**; easing always smooth, never "elastic"; scroll-triggered animations fire once, never loop.
- Component-library responsibility split: base components → shadcn/ui; animation/transition logic → Motion; ambient backgrounds → React Bits; connective/flow diagrams → Magic UI; hover micro-interactions → Aceternity UI.
- Every interactive component must implement all states: default, hover, focus, active, disabled, loading, success, error.
- Each page section answers exactly one visitor question, never two (Hero → "who are you?", Ecosystem → "how does it work?", Portfolio → "what have you built?", CTA → "how do I hire you?").

## Non-negotiable quality bars

These gate "done" for any implemented feature (Capítulo 27):

- Lighthouse: Performance ≥ 95, Accessibility = 100, Best Practices = 100, SEO = 100.
- Core Web Vitals: LCP < 2.5s, INP < 200ms, CLS < 0.1, TTFB < 800ms.
- WCAG 2.2 AA compliance (full keyboard navigation, visible focus states, 4.5:1 text contrast, semantic HTML over ARIA/divs).
- No manual code edits required to manage portfolio content — it's CRM-driven via API (Capítulo 25); the website must never store project data locally.
- Private routes must be `noindex`/`nofollow` and excluded from the sitemap.
- No component over ~250 lines without justification; components decoupled/reusable; zero business logic mixed into presentation components (Capítulo 18).
